# Планирование: анализ, идентификация проблем и поиск решений
## Проблемы
- долгое исполнение заказов или вообще не выполняются – для B2B и B2С
- события из интернет магазина попадают на расчет цены в MES только после ручной активности через CRM API
- список заказов в MES медленно загружается не смотря на фильтры и пагинацию
- заказ берет сам оператор, можно распределять автоматически и равномерно 
- сложность расчета цены по 3D модели
- взаимодействие с внешними магазинами – API часть MES системы
- каждое приложение и БД представлены в 1 инстансе
- одна команда работает над всеми сервисами
- не автоматизировано интеграционное и E2E тестирование

## Инициативы
1. Настроить мониторинг, логирование и трейсинг для подтверждения гипотез (поможет приоритезировать пункты ниже, оценить влияние)
2. Кэширование заказов в MES (решает проблему выдачи списка операторам)
3. Вынести расчет стоимости из MES в отдельный сервис – это происходит чаще, чем заказы поступают на изготовление (можем масштабировать отдельно + разумно в рамках DDD)
4. Отправлять асинхронно запросы к сервису «расчета стоимости» напрямую из «интернет-магазина»
5. Оптимизация расчета стоимости
6. Реализовать шардирование для БД MES – для ускорения загрузки заказов 
7. Производство заказа назначается автоматически на оператора с меньшей нагрузкой
8. Запланировать масштабирование сервисов – несколько реплик, доработки до stateless
9. Выделить (нанять) отдельные команды разработки для MES, CRM, Web-приложения магазина
10. Нанять automation QA и автоматизировать тестирование, сократив T2M для релизов

## Архитектура TOBE 1Y
На основе проблем и инициатив предлагается следующая архитектура на горизонте 6-12 месяцев - файл `Александрит TOBE 1Y.png`
Формат – PNG, можно увеличить.

## Негативный сценарий (успеваем 3 ключевые инициативы)
1. Кэширование для заказов в MES
2. Выделение расчета стоимости в отдельный сервис и оптимизация его логики
3. Выделение взаимодействия B2B в отдельный сервис (external API)
