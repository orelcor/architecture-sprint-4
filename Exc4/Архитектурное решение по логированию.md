# Архитектурное решение по логированию

## Мотивация
Логирование является необходимым для разбора инцидентов, которые регулярно возникают в системе. Иначе для сотрудника тех. поддержки или разработчиков практически не реально ошибки формата «со слова клиента» трансформировать в технологические гипотезы.

## Предлагаемое решение
Необходимо использовать следующие уровни логирования:
- INFO - базовая информация по работе приложения
- ERROR - ошибки
- DEBUG – технические подробности

Согласно схеме ASIS - необходимо дополнительно развернуть Logstash для сбора логов из сервисов Александрита. Логи могут записываться напрямую, либо собираться из файлов на воркерах приложения агентом (например Filebeat). Elastic Search для построения индексов над логами и Kibana как средство доступа визуализации.

Предлагаемая глубина хранения - 7 дней в Elastic/Logstash. Далее архивирование в холодное хранилище в виде файлов - в целях экономии КТС.

С точки зрения безопасности достаточно ролевой модели, реализованной в рамках стека ELK, интегрированного с LDAP-системой организации.

### На базе основных событий и сущностей можем писать следующие логи:
#### Сущность - заказ (INFO):
1. ID заказа — уникальный идентификатор заказа.
2. Статус заказа — текущий статус заказа (например, INITIATED, SUBMITTED и т.д.).
3. Время изменения статуса — временная метка, когда произошла смена статуса.
4. ID пользователя — идентификатор пользователя, который инициировал или изменил заказ.

#### Сущность - производство (INFO):
1. ID процесса — уникальный идентификатор производственного процесса.
2. ID заказа — идентификатор заказа, связанного с производственным процессом.
3. Статус процесса — текущий статус производственного процесса (например, MANUFACTURINGSTARTED, MANUFACTURINGCOMPLETED).
4. Время изменения статуса — временная метка, когда статус производственного процесса был изменён.

#### Cущность - упаковка и доставка (INFO):
1. ID упаковки — уникальный идентификатор упаковки заказа.
2. ID заказа — идентификатор заказа, связанного с упаковкой.
3. Статус упаковки — текущий статус упаковки (например, PACKAGING, SHIPPED).
4. Время изменения статуса — временная метка, когда статус упаковки был изменён.

#### Сущность - закрытие заказа (INFO)
1. ID заказа — уникальный идентификатор закрываемого заказа.
2. Причина закрытия — причина завершения заказа (например, доставка завершена, отмена пользователем).
3. Время закрытия — временная метка, когда заказ был закрыт.
4. ID оператора — идентификатор оператора или пользователя, который закрыл заказ (если применимо).

#### Ошибки, которые отловлены в приложении (ERROR):
1. ID ошибки — уникальный идентификатор для каждой записи об ошибке.
4. Сообщение об ошибке — текстовое сообщение, описывающее суть ошибки.
5. Стек вызовов — информация о стеке вызовов в момент возникновения ошибки (если применимо).
6. ID пользователя — идентификатор пользователя, который мог быть связан с ошибкой (если применимо).
7. ID заказа — идентификатор заказа, в контексте которого произошла ошибка (если применимо).
8. Модуль/компонент — название модуля или компонента системы, где произошла ошибка (например, API, База данных, UI и т.д.).
9. IP-адрес — IP-адрес клиента, с которого была инициирована операция (если применимо).

#### Cущность - пользователь (DEBUG):
1. ID пользователя — уникальный идентификатор пользователя.
2. Имя пользователя — имя или логин пользователя.
3. Email пользователя — адрес электронной почты для связи.
4. Время последней активности — временная метка последнего действия пользователя в системе.

#### Cущность - 3D-модель (DEBUG):
1. ID модели — уникальный идентификатор загруженной 3D-модели.
2. Формат файла — формат загруженного файла (например, .stl, .obj и т.д.).
3. Размер файла — размер загруженного файла в байтах.
4. Время загрузки — временная метка, когда файл был загружен.

## Аналитика
Встроенные средства Kibana можно использовать для регулярной аналитике по работе приложения. На их основе могут быть выделены дополнительные метрики:
1. Количество созданных заказов
2. Количество ошибок на каждом этапе производственного процесса
3. Количество незакрытых заказов
4. Размер файла 3D-модели

Также на основе получившихся метрик можно настроить дополнительный алертинг, опять же нативно через ELK.
