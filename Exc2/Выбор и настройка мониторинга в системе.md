# Выбор и настройка мониторинга в системе «Александрит»

## Мотивация
Мониторинг необходим для определения типовой нагрузки на синхронные и асинхронные интерфейсы как внутри системы, так и для внешнего API.
Это позволит:
1.	Оценить текущую нагрузку 
2.	Выявить узкие места с высокой нагрузкой
3.	Реагировать на проблемы в системе проактивно (с учетом кратно растущей нагрузки от внешних)

## Выбор подхода к мониторингу
В первую очередь мониторинг необходим для интерфейсов MES. Причина – эта подсистема является узким местом на данный момент. Так как речь идет про мониторинг интерфейсов – подходит подход RED.
Request Rate – поможет следить за растущей нагрузкойErrors – поможет настроить алертинг на ошибкиDuration – позволит оценить задержки при взаимодействиях
Также в рамках оценки необходимости масштабирования и оценки утилизации – предлагаются метрик в концепции USE (в первую очередь Utitlization)

## Метрики
1.	Number of requests (RPS) for MES API – нагрузка на сервис
2.	Number of requests (RPS) per user for MES API – перекос нагрузки (топ-потребителей)
3.	Number of HTTP 200 for MES API – количество успешных обращений
4.	Number of HTTP 500 for MES API – количество ошибок (визуализировать пропорцию относительно успешных запросов)
5.	Response time (latency) for MES API – для оценки задержки
1.	Size of MES db instance – для оценки необходимости масштабирования хранилища
2.	Number of connections for MES db instance – поиск ошибочно не закрытых сессий (риск, учитывая большое количество потребителей
3.	Number of message in flight in RabbitMQ – мониторинг здоровья очереди
4.	Number of dead-letter-exchange letters in RabbitMQ – мониторинг здоровья очереди
6.	CPU % for MES API – для оценки вычислительной нагрузки (скорее всего подтвердит необходимость выделить «расчет цены» в отдельный сервис)

## План действий
1.	Развернуть Prometheus для хранения метрик и Grafana для визуализации (можно в Сloud)
2.	Настроить http-экспортеры для интерфейсов
3.	Сконфигурировать Prometheus на созданные агенты
4.	Выделить пограничные значения нагрузки и настроить алертинг
5.	Построить дашборды в Grafana на Prometheus в качества Data Source
