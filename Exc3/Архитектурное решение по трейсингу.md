# Архитектурное решение по трейсингу

## Мотивация
Трейсинг позволит в случае ошибок в распределенных транзакциях (по заказам) определять на каком уровне произошла ошибка. Таким образом время обработки инцидентов значительно сократится.
Это повлияет на следующие метрики:
1. Сократит T2M продукта (на выпуск фикс-релизов)
2. Сократит среднее время закрытия тикета поддеркжи
3. Уменьшит количество инцидентов
4. В среднесрочной перспективе приведет к своевременному выявлению ошибок, влияющих на количество успешных заказов – сокращение ущерба имиджу компании

## Предлагаемое решение
Трейсингом предлагается покрыть бэкенды-компоненты, вовлеченные в распределенную транзакцию: Shop API, CRM API, MES API и «расчет стоимости» (уделить внимание последним в виду частых ошибок)

План:
1. Развернуть Jaeger и Jaeger UI на серверах компании
2. С помощью SDK OpenTelemetry реализовать отправку логов с указанием сквозного идентификатора заказа в процессор (коллектор jaeger)
3. Через Jaeger UI сотрудник второй линии тех. поддержки сможет выявить проблемное место в заказе – актуализирует статусы или заводит баг на команду разработки
4. Настраивается алертинг на основе длительности вызовов

## Компромиссы
Потенциально трейсинг полезен во всех бэкенд-компонентах, однако это может быть трудоемко. Например, в External API предлагается этого не делать, т.к. через него заказ для B2B создается и вычитывается его статусы.
## Безопасность
Данные не содержат тайны и обезличены.
Однако, данные все равно недоступны во внешнем контуре – только для аутентифицированных в системе сотрудников. Дополнительно реализуется авторизация (ролевая модель), в которой доступ к трейсингу есть только у поддержки и разработчиков.
